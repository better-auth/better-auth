## Overview

Better Auth Infrastructure supports:

- **SAML 2.0** - Industry standard SSO protocol
- **OIDC** - OpenID Connect for modern applications
- **Domain-based routing** - Automatic IdP selection by email domain

|---------|---------|-----|----------|------------|
| SSO/SAML | - | - | Yes | Yes |

SSO is available on Business and Enterprise plans.

## How SSO Works

```
┌─────────┐     ┌─────────────────┐     ┌─────────────┐
│  User   │────>│  Your App       │────>│  Identity   │
│         │     │  (Better Auth)  │     │  Provider   │
│         │<────│                 │<────│  (Okta,     │
│         │     │                 │     │  Azure AD)  │
└─────────┘     └─────────────────┘     └─────────────┘

1. User enters email
2. App detects SSO domain
3. User redirected to IdP
4. User authenticates at IdP
5. IdP sends assertion back
6. User signed in to app
```

## Setting Up SSO

### Creating an SSO Provider

1. Navigate to the organization
2. Go to the **SSO** tab
3. Click **Add SSO Provider**
4. Choose provider type:
   - **SAML 2.0**
   - **OIDC (OpenID Connect)**

### SAML 2.0 Configuration

#### Your App (Service Provider) Settings

Provide these to your customer's IT team:

| Setting | Value |
|---------|-------|
| Entity ID | `https://yourapp.com/api/auth/sso/saml/{org_id}` |
| ACS URL | `https://yourapp.com/api/auth/sso/saml/callback` |
| SLO URL | `https://yourapp.com/api/auth/sso/saml/logout` (optional) |
| Name ID Format | Email Address |

#### Identity Provider Settings

Collect these from your customer:

| Setting | Description |
|---------|-------------|
| IdP Entity ID | Unique identifier for the IdP |
| SSO URL | Where to send auth requests |
| Certificate | X.509 certificate for signature validation |

Or use **Metadata URL** if available - Better Auth will fetch settings automatically.

### OIDC Configuration

#### Your App Settings

Provide these to your customer:

| Setting | Value |
|---------|-------|
| Redirect URI | `https://yourapp.com/api/auth/sso/oidc/callback` |
| Logout URI | `https://yourapp.com/api/auth/sso/oidc/logout` (optional) |

#### Identity Provider Settings

Collect these from your customer:

| Setting | Description |
|---------|-------------|
| Issuer URL | OpenID Connect issuer |
| Client ID | Application client ID |
| Client Secret | Application client secret |

Or use **Discovery URL** (`.well-known/openid-configuration`) for automatic setup.

## Domain Verification

Before SSO can be used, verify domain ownership:

### DNS Verification

1. Go to the SSO provider settings
2. Click **Verify Domain**
3. Add the provided TXT record to your DNS:
   ```
   _betterauth.yourdomain.com TXT "verify=abc123..."
   ```
4. Click **Check Verification**

### HTTP Verification

Alternative to DNS verification:

1. Download the verification file
2. Host it at:
   ```
   https://yourdomain.com/.well-known/better-auth-verification.txt
   ```
3. Click **Check Verification**

### Verification Status

| Status | Description |
|--------|-------------|
| Pending | Awaiting verification |
| Verified | Domain ownership confirmed |
| Failed | Verification check failed |

## Testing SSO

Before enabling for all users:

1. Go to SSO provider settings
2. Click **Test Connection**
3. You'll be redirected to the IdP
4. Authenticate with a test account
5. Verify the test results:
   - User attributes mapped correctly
   - Email matches expected format
   - Groups/roles synced (if configured)

## Enabling SSO

### SSO Modes

| Mode | Description |
|------|-------------|
| Optional | Users can use SSO or other auth methods |
| Required | Users with matching domain must use SSO |
| Exclusive | Only SSO is allowed for the organization |

### Enabling for Users

1. Ensure domain is verified
2. Set SSO mode (optional/required)
3. Users with matching email domains will see SSO option
4. Or be automatically redirected (if required)

## User Flow

### Optional SSO

```
1. User enters email
2. App shows login options:
   - Continue with SSO
   - Sign in with password
3. User chooses
```

### Required SSO

```
1. User enters email
2. Domain matches SSO configuration
3. User automatically redirected to IdP
4. No password option shown
```

## Attribute Mapping

Map IdP attributes to user fields:

### Default Mappings

| User Field | SAML Attribute | OIDC Claim |
|------------|----------------|------------|
| email | NameID or email | email |
| name | displayName | name |
| firstName | givenName | given_name |
| lastName | sn | family_name |

### Custom Mappings

Configure custom attribute mappings:

1. Go to SSO provider settings
2. Click **Attribute Mapping**
3. Map IdP attributes to user fields:
   ```
   IdP Attribute → User Field
   http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress → email
   http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name → name
   ```
4. Save changes

## Just-in-Time Provisioning

Automatically create user accounts on first SSO login:

### Enabled (Default)

- User authenticates via IdP
- Account created in your app
- User added to organization
- Profile populated from IdP attributes

### Disabled

- User must exist before SSO login
- Pre-provision users via API or directory sync
- Useful for strict user management

## Group/Role Sync

Sync IdP groups to organization roles:

1. Configure group claim in IdP
2. Go to **Attribute Mapping**
3. Map group attribute:
   ```
   groups → organizationRole
   ```
4. Configure role mappings:
   ```
   IdP Group → App Role
   Admins → admin
   Users → member
   ```

## API Integration

### Create SSO Provider

```ts
await authClient.organization.createSsoProvider({
  organizationId: "org_123",
  type: "saml",
  name: "Okta",
  domain: "company.com",
  metadata: {
    entityId: "https://company.okta.com/...",
    ssoUrl: "https://company.okta.com/sso/saml",
    certificate: "-----BEGIN CERTIFICATE-----...",
  },
});
```

### Verify Domain

```ts
await authClient.organization.verifySsoProviderDomain({
  organizationId: "org_123",
  providerId: "sso_123",
});
```

### List SSO Providers

```ts
const providers = await authClient.organization.listSsoProviders({
  organizationId: "org_123",
});
```

## Troubleshooting

### Common Issues

**"SSO provider not found"**
- Verify domain is added and verified
- Check email domain matches exactly

**"Invalid SAML response"**
- Verify certificate is correct
- Check ACS URL matches exactly
- Ensure clock sync between systems

**"User not found"**
- Enable JIT provisioning
- Or pre-create user accounts

**"Attribute mapping failed"**
- Check IdP is sending required attributes
- Verify attribute names match configuration

### Debug Mode

Enable SSO debug mode for detailed logs:

1. Go to SSO provider settings
2. Enable **Debug Mode**
3. Attempt authentication
4. Review logs for detailed error information

## Security Considerations

1. **Use HTTPS** - Always use HTTPS for SSO endpoints
2. **Verify certificates** - Validate IdP certificates
3. **Limit domains** - Only verify domains you own
4. **Monitor logs** - Watch for failed SSO attempts
5. **Regular audits** - Review SSO configurations periodically
