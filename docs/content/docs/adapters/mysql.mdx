MySQL is a popular open-source relational database management system (RDBMS) that is widely used for building web applications and other types of software. It provides a flexible and scalable database solution that allows for efficient storage and retrieval of data.
Read more here: [MySQL](https://www.mysql.com/).

## Example Usage

Make sure you have MySQL installed and configured.
Then, you can connect it straight into Better Auth.

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { createPool } from "mysql2/promise";

export const auth = betterAuth({
  database: createPool({
    host: "localhost",
    user: "root",
    password: "password",
    database: "database",
    timezone: "Z", // Important to ensure consistent timezone values
  }),
});
```

<Callout>
  For more information, read Kysely's documentation to the
  [MySQLDialect](https://kysely-org.github.io/kysely-apidoc/classes/MysqlDialect.html).
</Callout>

## Joins (Experimental)

Database joins is useful when Better-Auth needs to fetch related data from multiple tables in a single query.
Endpoints like `/get-session`, `/get-full-organization` and many others benefit greatly from this feature,
seeing upwards of 2x to 3x performance improvements depending on database latency.

The Kysely MySQL dialect supports joins out of the box since version `1.4.0`.

To enable this feature, you need to set the `experimental.joins` option to `true` in your auth configuration.

```ts title="auth.ts"
import { betterAuth } from "better-auth";

export const auth = betterAuth({
  experimental: { joins: true }
});
```

<Callout type="warn">
  It's possible that you may need to run migrations after enabling this feature.
</Callout>

## ID Generation on MySQL

MySQL does not support the `RETURNING` clause that PostgreSQL and SQLite use to retrieve inserted rows. Because of this, how you configure ID generation matters on MySQL.

### Recommended Approaches

* **Use Better Auth's default ID generation** (no configuration needed) — Better Auth generates IDs before inserting, so it always knows which row was inserted.
* **Use a custom `generateId` function** — provide your own ID generator via `advanced.database.generateId`.
* **Use `"serial"` for auto-increment integer IDs** — set `advanced.database.generateId` to `"serial"`.

### What to Avoid

<Callout type="error">
  **Do not** set `advanced.database.generateId` to `false` on MySQL if your primary keys use non-sequential ID schemes (UUIDv4, CUID2, ULID, or any non-monotonic values generated by the database or Drizzle's `$defaultFn()`).

  When `generateId` is `false`, Better Auth does not know the ID of the newly inserted row. On MySQL, the adapter must re-query the database to find it. If your IDs are not sequential, this re-query can return **the wrong row** — potentially logging users into other users' accounts.
</Callout>

**If you use the Drizzle adapter** with `drizzle-orm` **0.32.0+**, the adapter uses `$returningId()` to safely retrieve the inserted ID. If your `drizzle-orm` version is older or `$returningId()` fails, the adapter will throw an error rather than risk returning the wrong row.

**If you use the Kysely adapter** (the default for MySQL), only **auto-increment integer primary keys** are supported when `generateId` is `false`. Non-auto-increment PKs will cause the adapter to throw an error.

If you need database-generated non-sequential IDs on MySQL, use a custom `generateId` function that generates the ID in your application code:

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { createId } from "@paralleldrive/cuid2";

export const auth = betterAuth({
  database: createPool({ /* ... */ }),
  advanced: {
    database: {
      generateId: () => createId(),
    },
  },
});
```

For more details on ID generation options, see the [Database documentation](/docs/concepts/database#id-generation).
