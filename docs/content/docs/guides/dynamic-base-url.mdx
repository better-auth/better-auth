Some deployments answer on more than one hostname:

* custom domains like `myapp.com` and `www.myapp.com`
* preview deployments like `my-app-abc123.vercel.app`
* branch environments like `feature-branch.myapp.com`
* white-label tenants on their own domains, added at runtime

Better Auth handles this without running a separate instance per host. The key idea is a split between two things that used to be conflated:

* **`baseURL` is the canonical origin.** It anchors identity: the OAuth/OIDC issuer, JWT `iss`/`aud`, the social-login `redirect_uri`, and the Passkey relying-party id. It does not change per request.
* **`trustedOrigins` is the set of hosts you also serve.** When a request arrives on one of these hosts, cookies and self-referential links (email verification, magic links, password reset) follow that host. Identity still uses `baseURL`.

This keeps preview deploys and tenant domains working for cookies and links, while tokens, discovery documents, and passkeys stay stable across hosts.

## Basic Setup

Set a fixed `baseURL` and list the extra hosts in `trustedOrigins`:

```ts title="auth.ts"
import { betterAuth } from "better-auth"

export const auth = betterAuth({
	baseURL: "https://myapp.com",
	trustedOrigins: [
		"https://www.myapp.com",
		"https://*.vercel.app",
	],
})
```

When a request comes in, Better Auth derives the host from the `host` header or the request URL (and from `x-forwarded-host` when you set `advanced.trustedProxyHeaders: true`). If the resulting origin is in `trustedOrigins`, cookies and links use it; otherwise they fall back to `baseURL`.

## Common Deployment Patterns

### Vercel preview deployments

```ts title="auth.ts"
export const auth = betterAuth({
	baseURL: "https://myapp.com",
	trustedOrigins: ["https://*.vercel.app"],
})
```

### Multiple production domains

```ts title="auth.ts"
export const auth = betterAuth({
	baseURL: "https://myapp.com",
	trustedOrigins: [
		"https://myapp.co.uk",
		"https://myapp.eu",
	],
})
```

### White-label tenants (domains from a database)

When valid domains are not known at deploy time, pass a function. It runs per request and can read the request to decide which origins are valid.

```ts title="auth.ts"
export const auth = betterAuth({
	baseURL: "https://app.myplatform.com",
	trustedOrigins: async (request) => {
		const host = request?.headers.get("host")
		if (!host) return []
		const tenant = await findTenantByDomain(host)
		return tenant ? [`https://${host}`] : []
	},
})
```

<Callout type="warn">
  With the function form you own the allowlist. Only return origins you control; an attacker-supplied host that you return becomes a valid cookie and link target. Cache the lookup if it hits a database on every request.
</Callout>

## Cookies Across Subdomains

With `crossSubDomainCookies` enabled and no explicit `domain`, the cookie domain follows the host the request arrived on. A request to `auth.example1.com` sets cookies for `auth.example1.com`; a request to `auth.example2.com` sets cookies for `auth.example2.com`.

```ts title="auth.ts"
export const auth = betterAuth({
	baseURL: "https://auth.example1.com",
	trustedOrigins: ["https://auth.example2.com"],
	advanced: {
		crossSubDomainCookies: { enabled: true },
	},
})
```

To pin one shared parent domain instead, set it directly:

```ts title="auth.ts"
export const auth = betterAuth({
	baseURL: "https://auth.example1.com",
	advanced: {
		crossSubDomainCookies: {
			enabled: true,
			domain: ".example.com",
		},
	},
})
```

See [Cookies](/docs/concepts/cookies#cross-subdomain-cookies) for the full cookie behavior.

## Identity Stays on the Canonical Origin

Identity-bearing values deliberately do **not** follow the request host, because their specs bind them to a stable origin:

* **OAuth/OIDC issuer and JWT `iss`/`aud`** stay on `baseURL`, so tokens and discovery documents agree regardless of which host served them.
* **Social-login `redirect_uri`** stays on `baseURL`, because upstream providers (Google, etc.) reject unregistered redirect URIs. To send users back to the host they started on, use [OAuth Proxy](/docs/plugins/oauth-proxy).
* **Passkey relying-party id** stays on `baseURL`, because credentials are bound to the rp id at registration and would break if it changed with an ephemeral preview host.

Identity is fixed per Better Auth instance. If you genuinely need a separate identity per tenant (a distinct issuer or Passkey rp id per tenant domain), run one Better Auth instance per tenant, each with its own `baseURL`, and route requests to the right instance by host. `jwt.issuer`, the OAuth Provider issuer, the Passkey `rpID`, and `oAuthProxy.productionURL` are static per instance; they do not vary by request.

## Inside database hooks

Database hooks receive the endpoint context as their second argument. Read the canonical origin from `context.context.baseURL`, or the serving origin (the host this request arrived on) with `getRequestBaseURL`:

```ts title="auth.ts"
import { getRequestBaseURL } from "better-auth/api"

export const auth = betterAuth({
	baseURL: "https://myapp.com",
	trustedOrigins: ["https://*.vercel.app"],
	databaseHooks: {
		user: {
			create: {
				after: async (user, context) => {
					// canonical: "https://myapp.com/api/auth"
					const canonical = context?.context.baseURL
					// serving host for this request, e.g. "https://preview.vercel.app/api/auth"
					const servingURL = context ? getRequestBaseURL(context) : undefined
				},
			},
		},
	},
})
```

`context` is `null` when the hook runs outside a request (such as a CLI migration), so guard for it.

## Security Model

1. The request host comes from the `Host` header or request URL. `x-forwarded-host` is honored only when you opt in with `advanced.trustedProxyHeaders: true`, which a reverse proxy that exposes the public host that way must set.
2. The derived origin is honored for cookies and links only when it is in `trustedOrigins`; otherwise Better Auth falls back to the canonical `baseURL`.
3. `trustedOrigins` is also the CSRF and redirect allowlist, so listing a host opts it into the same validation everywhere.
