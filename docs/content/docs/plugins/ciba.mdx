`CIBA` `AI Agents` `CLI` `Headless`

The CIBA plugin implements [Client Initiated Backchannel Authentication (CIBA)](https://openid.net/specs/openid-client-initiated-backchannel-authentication-core-1_0.html), enabling authentication flows where an agent or service authenticates on behalf of a user who approves out-of-band — via email, SMS, push notification, or any custom channel.

This is ideal for:
- **AI agents** that need to act on behalf of a user
- **CLI tools** that authenticate without a browser redirect
- **Headless services** that request user authorization asynchronously
- **IoT devices** with limited or no UI

**Key Features**:

- **Poll, ping, and push** token delivery modes
- **`authorization_details`** (Rich Authorization Requests, RFC 9396) for structured consent
- **`id_token_hint`** with full JWT signature validation (JWKS asymmetric + HS256 fallback)
- **`requested_expiry`** for client-controlled request lifetimes
- **Resource indicators** forwarded to the token pipeline
- **HTTPS enforcement** on push/ping notification endpoints (CIBA §10.3)
- **Replay protection** — requests are consumed on token issuance
- **Rate limiting** with `slow_down` and progressive interval increase

Requires the [`@better-auth/oauth-provider`](/docs/plugins/oauth-provider) and [`jwt`](/docs/plugins/jwt) plugins.

## How It Works

1. **Agent requests auth**: The agent calls `/oauth2/bc-authorize` with client credentials and a `login_hint` (user's email) or `id_token_hint`
2. **User is notified**: Your `sendNotification` callback delivers an approval link to the user
3. **User approves/denies**: The user visits the approval page and approves or rejects
4. **Agent gets tokens**: The agent polls `/oauth2/token` (poll mode), receives a ping notification then polls (ping mode), or receives tokens directly at a callback URL (push mode)

## Basic Usage

### Agent: Initiate Authentication

The agent sends a backchannel authorize request with client credentials and a user hint:

```ts
const response = await fetch("https://your-app.com/api/auth/oauth2/bc-authorize", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Basic ${btoa(`${clientId}:${clientSecret}`)}`,
  },
  body: JSON.stringify({
    scope: "openid profile",
    login_hint: "user@example.com",
    binding_message: "Sign in to My CLI Tool",
  }),
});

const { auth_req_id, expires_in, interval } = await response.json();
```

The response contains:
- `auth_req_id` — unique request identifier (~190-bit entropy)
- `expires_in` — request lifetime in seconds
- `interval` — minimum polling interval in seconds (poll/ping mode only)

### User: Approve the Request

When the user clicks the approval link, your app shows an approval page. Use `auth_req_id` from the URL to fetch request details:

```tsx title="app/ciba/approve/page.tsx"
export default function ApprovalPage() {
  const searchParams = useSearchParams();
  const authReqId = searchParams.get("auth_req_id");
  const [request, setRequest] = useState(null);

  useEffect(() => {
    authClient.ciba.verify({
      query: { auth_req_id: authReqId },
    }).then(({ data }) => setRequest(data));
  }, [authReqId]);

  const handleApprove = () =>
    authClient.ciba.authorize({ auth_req_id: authReqId });

  const handleReject = () =>
    authClient.ciba.reject({ auth_req_id: authReqId });

  if (!request) return <p>Loading...</p>;

  return (
    <div>
      <h2>Sign-in Request</h2>
      <p>Client: {request.client_name}</p>
      <p>Message: {request.binding_message}</p>
      <p>Scopes: {request.scope}</p>
      <p>Expires: {new Date(request.expires_at).toLocaleString()}</p>
      <button onClick={handleApprove}>Approve</button>
      <button onClick={handleReject}>Deny</button>
    </div>
  );
}
```

<Callout type="warn">
  Users must be authenticated before they can approve or deny requests. Redirect unauthenticated users to the login page with a return URL back to the approval page.
</Callout>

### Agent: Poll for Tokens

After initiating the request, the agent polls until the user responds:

```ts
async function pollForToken(authReqId: string, interval: number) {
  let pollingInterval = interval;

  while (true) {
    await new Promise((r) => setTimeout(r, pollingInterval * 1000));

    const response = await fetch("https://your-app.com/api/auth/oauth2/token", {
      method: "POST",
      headers: {
        "Content-Type": "application/x-www-form-urlencoded",
        "Authorization": `Basic ${btoa(`${clientId}:${clientSecret}`)}`,
      },
      body: new URLSearchParams({
        grant_type: "urn:openid:params:grant-type:ciba",
        auth_req_id: authReqId,
      }),
    });

    const data = await response.json();

    if (data.access_token) {
      return data; // { access_token, token_type, id_token, scope, ... }
    }

    switch (data.error) {
      case "authorization_pending":
        break; // Continue polling
      case "slow_down":
        pollingInterval += 5;
        break;
      case "access_denied":
        throw new Error("User denied the request");
      case "expired_token":
        throw new Error("Request expired");
    }
  }
}
```

<Callout type="info">
  The `/oauth2/token` endpoint is the existing OAuth provider token endpoint. The CIBA plugin hooks into it when the grant type is `urn:openid:params:grant-type:ciba`. The `/oauth2/` prefix comes from the OAuth provider's route naming.
</Callout>

## Authorization Details (RAR)

For structured consent beyond free-text `binding_message`, pass `authorization_details` per [RFC 9396](https://datatracker.ietf.org/doc/html/rfc9396). This lets the user see exactly what they're approving:

```ts
const response = await fetch("/api/auth/oauth2/bc-authorize", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Basic ${btoa(`${clientId}:${clientSecret}`)}`,
  },
  body: JSON.stringify({
    scope: "openid",
    login_hint: "user@example.com",
    binding_message: "Purchase: Sony WH-1000XM5",
    authorization_details: JSON.stringify([{
      type: "purchase",
      merchant: "Aether AI",
      item: "Sony WH-1000XM5",
      amount: { currency: "USD", value: "378.45" },
    }]),
  }),
});
```

The `authorization_details` are:
- Validated as JSON at request time (malformed JSON returns `400`)
- Stored with the CIBA request
- Returned from the `/ciba/verify` endpoint (parsed) for the approval UI
- Available in the token response

## Delivery Modes

### Poll (default)

The agent polls `/oauth2/token` at the interval specified in the bc-authorize response. The server enforces rate limiting — polling too fast returns `slow_down` and increases the required interval by 5 seconds.

### Ping

The server sends a notification to the client's endpoint when the user responds, signaling the agent to poll for tokens. This reduces unnecessary polling.

```ts
ciba({
  deliveryModes: ["ping", "poll"],
  sendNotification: async (data) => { /* ... */ },
  resolveClientNotificationEndpoint: async (clientId, ctx) => {
    return "https://my-agent.example.com/ciba-callback";
  },
})
```

The agent must include `client_notification_token` in the bc-authorize request.

### Push

The server delivers tokens directly to the client's notification endpoint. No polling needed.

```ts
ciba({
  deliveryModes: ["push", "poll"],
  sendNotification: async (data) => { /* ... */ },
  resolveClientNotificationEndpoint: async (clientId, ctx) => {
    return "https://my-agent.example.com/ciba-callback";
  },
})
```

The agent must include `client_notification_token` in the bc-authorize request. Push mode requires both a notification token and endpoint. The response omits `interval` since polling is not used.

<Callout type="warn">
  Push and ping notification endpoints must use HTTPS per CIBA §10.3. Loopback addresses (`localhost`, `127.0.0.1`, `[::1]`) are exempt for local development.
</Callout>

## ID Token Hint

Instead of `login_hint`, agents can pass a previously issued `id_token_hint` to identify the user. The plugin validates:
- JWT signature (JWKS asymmetric keys first, HS256 fallback)
- `aud` claim matches the requesting `client_id`
- `iss` claim matches the server's issuer
- Token is not expired

```ts
const response = await fetch("/api/auth/oauth2/bc-authorize", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Basic ${btoa(`${clientId}:${clientSecret}`)}`,
  },
  body: JSON.stringify({
    scope: "openid",
    id_token_hint: previousIdToken,
  }),
});
```

<Callout type="info">
  Only one hint may be provided per request. Sending both `login_hint` and `id_token_hint` (or `login_hint_token`) returns `400 invalid_request`.
</Callout>

## Notification Callback

The `sendNotification` callback is how you deliver the approval link to the user. You decide the channel:

```ts
ciba({
  sendNotification: async (data, request) => {
    // data.userId           — user ID to notify
    // data.authReqId        — unique request identifier
    // data.approvalUrl      — URL the user visits to approve/deny
    // data.clientName       — name of the requesting OAuth client
    // data.scope            — requested scopes
    // data.bindingMessage   — optional context message
    // data.authorizationDetails — parsed RAR object (if provided)

    await sendEmail({
      to: await lookupEmail(data.userId),
      subject: data.bindingMessage || "Sign-in request",
      body: `
        An application "${data.clientName}" is requesting access.

        ${data.bindingMessage ? `Message: ${data.bindingMessage}` : ""}
        Scopes: ${data.scope}

        Approve: ${data.approvalUrl}
      `,
    });
  },
})
```

The callback is fire-and-forget — failures are silently ignored so they don't block the bc-authorize response.

## User Resolution

By default, the plugin resolves users from `login_hint` by email lookup via the internal adapter. You can customize this:

```ts
ciba({
  sendNotification: async (data) => { /* ... */ },
  resolveUser: async (loginHint, ctx) => {
    // Custom lookup — by phone, username, external ID, etc.
    const user = await db.users.findOne({ phone: loginHint });
    return user; // Must return { id: string } or null
  },
})
```

## Endpoints

### Backchannel Authorize

Initiate a CIBA backchannel authentication request. Accepts `application/json` or `application/x-www-form-urlencoded`. Client credentials can be provided via HTTP Basic `Authorization` header instead of body parameters.

     * Space-separated scopes (must include "openid")

    /**
     * Token containing user hint (exactly one hint required)

     * Bearer token for push/ping callbacks

## Error Handling

The CIBA flow defines specific error codes:

| Error Code | Description |
|------------|-------------|
| `authorization_pending` | User hasn't responded yet (continue polling) |
| `slow_down` | Polling too frequently (increase interval by 5s) |
| `expired_token` | The auth request has expired |
| `access_denied` | User denied the request |
| `invalid_grant` | Invalid, consumed, or push-mode `auth_req_id` |
| `invalid_scope` | Scope missing `openid` |
| `invalid_request` | Malformed request, multiple hints, or missing fields |
| `unknown_user_id` | Could not find user from hint |
| `invalid_client` | Invalid client credentials |

## Options

### Server

**sendNotification** (required): Callback to notify the user about the pending request. Receives `(data, request?)` where `data` contains `userId`, `authReqId`, `approvalUrl`, `clientName`, `scope`, `bindingMessage`, and `authorizationDetails`. Returns `Promise<void>`.

**deliveryModes**: Supported token delivery modes. Default: `["poll"]`. Can include `"poll"`, `"ping"`, and `"push"`.

**requestLifetime**: Auth request lifetime in seconds. Default: `300` (5 minutes).

**pollingInterval**: Minimum polling interval in seconds. Default: `5`.

**approvalPage**: URL path for the approval page. Default: `"/ciba/approve"`.

**resolveUser**: Custom user resolution from `login_hint`. Takes `(hint, ctx)` and returns `Promise<{ id: string } | null>`. Default: email lookup via the internal adapter.

**resolveClientNotificationEndpoint**: Resolve the notification endpoint for push/ping delivery. Takes `(clientId, ctx)` and returns `Promise<string | undefined>`.

**pushRetryAttempts**: Push delivery retry attempts with exponential backoff. Default: `3`.

### Client

No client-specific configuration options. The plugin adds the following methods:

- **ciba.verify()**: Get request details for the approval UI
- **ciba.authorize()**: Approve a pending request (requires authentication)
- **ciba.reject()**: Reject a pending request (requires authentication)

## Security Considerations

1. **TLS Required**: Push/ping notification endpoints must use HTTPS (loopback exempt for development)
2. **Client Authentication**: Agents authenticate via client credentials — HTTP Basic or body parameters. Client secrets are validated using the oauth-provider's configured storage method (hashed, encrypted, or plain)
3. **Unguessable Request IDs**: `auth_req_id` is 32 cryptographically random alphanumeric characters (~190 bits of entropy)
4. **Session Required**: Only authenticated users can approve or reject requests, and the session user must match the target user
5. **ID Token Validation**: `id_token_hint` is verified with full JWT signature checking — not trusted at face value
6. **Request Expiry**: Requests automatically expire (default 5 minutes). Clients can request shorter lifetimes via `requested_expiry` but not longer
7. **Polling Rate Limiting**: `slow_down` responses increase the required interval by 5 seconds per violation
8. **Replay Protection**: Approved requests are deleted after token issuance. A second poll returns `invalid_grant`
9. **Client Re-verification**: For push delivery, the client is re-verified as non-disabled before tokens are generated
