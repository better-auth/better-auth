# Better Auth 1.7 RC and the v1 maintenance plan

Better Auth 1.7.0 release candidate (RC) is now available for testing. This release candidate is the last step before we publish 1.7.0 as stable, and we want feedback from real projects before we make that call.

If you can, please try the release candidate in a development or staging environment and tell us what breaks, what feels unclear, and what still needs to land before 1.7.0 becomes stable.

The feedback discussion is the main place to report release-candidate issues. GitHub issues still work for isolated bugs, but the discussion helps us keep release feedback in one place while we decide what must change before stable.

## What we need from testers

1. Move a non-production project to the 1.7 release candidate.
2. Read the [1.7 upgrade guide](/docs/guides/1-7-upgrade-guide) for the parts of Better Auth you use.
3. Test the flows your users rely on, especially sign-in, account linking, session handling, SSO, OAuth provider flows, MCP integrations, custom adapters, and deployment behind a proxy.
4. Post your feedback in the discussion, including your Better Auth version, runtime, framework, database adapter, and the area you tested.

The most useful feedback is specific. If an integration no longer works, tell us which provider, framework, runtime, and adapter are involved. If an upgrade step is confusing, tell us where you got stuck. If a breaking change blocks a real production setup, tell us what the old setup depended on and what path you expected to have.

## 1.7 is the last feature-focused v1 release

After 1.7.0 becomes stable, Better Auth v1 will enter maintenance mode.

That means v1 will mainly receive security fixes, bug fixes, and compatibility fixes. We will avoid adding new features to v1 unless the change is part of maintaining the existing major version. Examples include an integration fix, a missing compatibility case that 1.7 should have covered, or a small API addition needed to unblock a correctness or security fix.

We do not expect to publish another feature-focused v1 minor release after 1.7.0. If a maintenance fix needs a breaking change, or if a specific v1 compatibility problem cannot be handled safely in a patch release, we may still publish another v1 minor for that purpose. In that case, the minor would be scoped to maintaining v1, not to opening a new feature track.

| After 1.7.0 stable                           | What to expect                                           |
| -------------------------------------------- | -------------------------------------------------------- |
| Security fixes                               | Yes                                                      |
| Bug fixes                                    | Yes                                                      |
| Compatibility fixes for existing v1 surfaces | Yes                                                      |
| New v1 features                              | Avoided by default                                       |
| More v1 minor releases                       | Not expected, except for maintenance cases that need one |
| v2 work                                      | Main product focus                                       |

This is not because v1 is being abandoned. It is because v1 has grown into a very broad framework, and continuing to add new surfaces there would make the next architecture harder to build and harder to explain.

## Why we are moving focus to v2

Better Auth now serves several different kinds of projects. Some teams only need sign-in and session handling. Some need a full user-management surface. Some need enterprise SSO. Some need Better Auth to act as an OAuth or OpenID Connect provider for other apps. Some need MCP and machine-client flows.

Those use cases should not all have to carry the same shape.

In Better Auth v2, we want the framework to stay familiar while becoming more modular. A small app should be able to install the parts it needs for sign-in and sessions without taking on the full user-management or provider-side model. A product that needs user management should be able to add that surface explicitly. A project that needs Better Auth as an identity provider should be able to install the provider-side pieces, storage, and adapters required for that job.

That direction requires deeper changes than a normal minor release can carry. The package boundaries, schema, and adapter model need to change, and some names and contracts need to become more honest about what they represent.

We are not ready to publish the full v2 design yet, and we do not want this post to become a promise about final APIs. The important part is the direction: Better Auth v2 will be more modular, more explicit about storage and identity boundaries, and better suited to the different ways people use auth today.

## What this means for your project

If you are using v1 today, the best next step is to test 1.7.0 RC. That gives you the latest v1 behavior, and it gives us the best chance to fix release-blocking issues before 1.7.0 becomes stable.

If your project depends on a v1 feature that still feels incomplete, bring that feedback to the discussion now. We especially want to hear about missing compatibility, integrations that should work but do not, migration blockers, and breaking changes that need a clearer path before the stable release.

After 1.7.0, we expect v1 to become quieter and more predictable. That stability is part of the plan: v1 remains maintained while v2 gets the deeper rewrite it needs.
