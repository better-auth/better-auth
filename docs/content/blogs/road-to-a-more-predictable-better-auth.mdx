Better Auth ships several packages from one repository, so every release needs coordination across packages, release tracks, and release notes, not just a version bump in one place. Before v1.6, fixes, additive improvements, large features, refactors, and breaking changes could all end up sharing the same release path. In practice, that meant a small fix could be ready now and still wait behind unrelated work.

Starting with v1.6, that changes.

## What changed for users

The new release process is designed around a simple goal: stable fixes should move faster.

In practical terms:

* `npm install better-auth` stays focused on stable releases
* `npm install better-auth@beta` tracks the next minor release in progress
* fixes no longer need to wait for unrelated feature work to settle

This gives Better Auth a clearer split between what is ready for stable users today and what needs a beta cycle first.

## How the new release tracks work

Better Auth now has 2 long-lived release tracks.

**`main`** is the stable line. It is where we ship bug fixes, security work, and additive improvements that do not require existing users to change code, configuration, or database schema.

**`next`** is the beta line. It is where we ship larger features, refactors, runtime or schema changes, and breaking changes.

The dividing line is migration risk. If a change asks existing users to update something, it should go through beta first.

That does not mean `main` is only for fixes. Some new capabilities can still land there if they are well-tested, non-breaking, and safe to adopt immediately. The point is to keep stable releases predictable.

## How stable and beta stay aligned

Splitting stable and beta only works if they stay connected.

Stable fixes still need to reach `next`, otherwise the beta branch stops reflecting the latest stable reality. The workflow we shipped handles that with batched sync PRs from `main` to `next`.

We chose that approach for a practical reason: it keeps contributor history intact and avoids constant force-pushes on `next`. It is not the theoretically cleanest Git history, but it is easier to operate and easier for contributors to follow.

For users, the outcome is simple: stable fixes can move quickly, and beta releases still include the latest stable work.

## What changed for contributors

The other major change is where release decisions now happen.

The old flow relied much more on maintainers deciding version bumps later in the process. The new flow moves that decision into the pull request itself with [Changesets](https://github.com/changesets/changesets).

When a PR changes published package code, it now carries a changeset with it. That makes the release intent visible during review instead of leaving it for the end.

The policy is enforced automatically:

* PRs that touch published packages need a changeset, unless they are explicitly marked with `skip-changeset`
* PRs targeting `main` are limited to `patch` changesets
* if a PR aimed at `main` carries a `minor` or `major` changeset, it is automatically retargeted to `next`
* changeset files are validated before merge

Once changesets accumulate, automation opens a **Version Packages** PR that shows exactly what would ship. A maintainer still reviews and merges that PR before publishing. Automation prepares the release, but humans still control the cadence.

## How the workflow helps contributors

One goal of this release work was to reduce friction for contributors.

The automation is meant to guide people toward the right path, not just block them with failing checks. That is why the workflow now includes:

* **Auto-retarget**, which moves PRs from `main` to `next` when the declared bump belongs on the beta track
* **`/changeset`**, which lets maintainers generate a changeset recommendation for contributors, including fork-based PRs
* **Automatic domain labels**, which organize PRs by architectural area for triage and release notes

The result is a contribution flow that is easier to understand. A safe fix can move through stable with clearer signals. A feature that belongs on beta gets redirected with an explanation.

## Release notes are clearer now

We also rebuilt how release notes are generated.

Previously, the output was closer to a raw changelog. It told you what changed, but it was harder to scan if you wanted the user-facing impact or the package affected.

The new pipeline has 2 layers:

* a deterministic step that finds the PRs in the release, groups them by package and area, and handles noisy cases like revert pairs
* an AI-assisted step that rewrites those entries into clearer user-facing language

The key design choice is that the AI step is an enhancement, not a dependency. If it fails, the structured release notes still exist. A release never has to stop because a rewrite step failed.

That package grouping is important. In the current `v1.6.0` release notes, changes are grouped under packages like `better-auth`, `@better-auth/sso`, and `@better-auth/oauth-provider` instead of being merged into one undifferentiated list. That makes the release easier to scan and keeps the changelog tied to the packages that actually moved together in the release.

This automation is part of the broader tooling we are building under the name **Better Release**, but the main user-facing outcome is simpler release communication.

## Releases are easier to trust, too

Speed only helps if the published package is still trustworthy.

Every npm publish now uses OIDC trusted publishing through GitHub Actions. There is no long-lived publish token sitting in CI waiting to be stolen. Each published package carries provenance that links it back to the source and workflow that produced it.

That matters because the biggest npm ecosystem failures have usually been credential and supply-chain problems. Commit signing and contributor attribution still matter, but for package consumers, provenance on the published artifact is the stronger guarantee.

You can verify a Better Auth package with:

```bash
npm audit signatures
```

## What this means for you

If you use Better Auth today, the new process should feel simpler:

* stable fixes can ship faster
* the beta channel is a clearer place to test upcoming work
* stable releases are less likely to mix urgent fixes with migration-heavy changes
* release notes are easier to scan by package and area
* published packages are easier to verify from source to registry

If you contribute to Better Auth, the workflow should also feel more predictable:

* release intent is declared in the PR, not guessed at the end
* the system helps correct branch targeting mistakes
* labels and changesets now feed directly into the release process

v1.6 was the first release to go through this pipeline. The process will keep evolving, but the direction is set: stable and beta now move on separate tracks, fixes have a faster path to users, and releases are easier to understand end to end.
