# Better Auth 1.7

Better Auth 1.7 is mainly an OAuth and OpenID Connect release. It matters most if your app signs users in with Google or Microsoft, lets other apps "Sign in with \[your app]", connects to a company identity provider, or powers an MCP server.

Three themes run through this release.

**Better Auth can act as a stronger identity provider.** If your app issues tokens to other apps, Better Auth now follows the OAuth and OpenID Connect rules more closely. DPoP tokens, single sign-out, forced re-login, per-API token rules, and safer token handling all work in a more standard way.

**Better Auth connects to more real identity providers.** Amazon Cognito, Microsoft Entra ID with certificate login, standards-based OpenID providers, and school-login providers like Clever now fit better.

**Better Auth is stricter by default.** Several security fixes make unsafe input fail earlier, stop trusting forwarded proxy headers unless you opt in, and replace database operations that could race under load.

Most projects can upgrade with the normal command. Some projects need a config change, a schema migration, or a short manual data step. The [1.7 upgrade guide](/docs/guides/1-7-upgrade-guide) has the exact steps.

## Upgrade impact at a glance

| If you use                                     | What to expect                                                                                              |
| ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Basic Better Auth setup                        | Usually just the upgrade command and any generated schema changes                                           |
| Social login, generic OAuth, One Tap, or SSO   | New login-client behavior, especially generic OAuth, identity-token verification, and Google One Tap        |
| `@better-auth/oauth-provider`                  | The largest set of changes: resources, DPoP, back-channel logout, `max_age`, and stricter protocol handling |
| MCP                                            | A package move to `@better-auth/mcp`, new imports, new endpoint paths, and a schema migration               |
| SAML or SCIM                                   | Node 20 or newer for SAML, stricter SAML and SSO checks, and a manual SCIM ownership cleanup                |
| Two-factor auth, magic links, or email OTP     | Stronger lockout behavior and safer handling for unverified credentials                                     |
| Custom adapters, storage, or rate-limit stores | New atomic methods are required                                                                             |
| Custom proxy or TLS termination                | Check how your app computes its public origin before relying on DPoP or identity-provider redirects         |

## OAuth vocabulary used below

| Term                    | Meaning in this release                                                                                                                                        |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Authorization server    | The service that issues access tokens. When other apps "Sign in with \[your app]", your app is the authorization server.                                       |
| OpenID Provider         | An authorization server that also proves who the user is. Google and Microsoft are OpenID Providers to your app; in 1.7 Better Auth can be one for other apps. |
| Relying party or client | The app receiving a login. When you add "Sign in with Google", your app is the relying party.                                                                  |

## Important changes

### Breaking changes

| Change                                           | What to do                                                                                                |
| ------------------------------------------------ | --------------------------------------------------------------------------------------------------------- |
| Protected resources replace `validAudiences`     | Move each audience into `resources`, link clients to resources, and run the schema migration.             |
| DPoP changes token verification                  | Rename the bearer-token helper, use the DPoP request verifier where needed, and run the schema migration. |
| Back-channel logout revokes session-bound tokens | Run the schema migration and expect tokens tied to a signed-out session to become inactive.               |
| Generic OAuth uses the social-provider path      | Update sign-in calls, link calls, callback URLs, and client plugins.                                      |
| One Tap needs a Google client ID                 | Set `oneTap({ clientId })` or `socialProviders.google.clientId`.                                          |
| The old `oidcProvider` plugin is removed         | Move provider setups to `@better-auth/oauth-provider`.                                                    |
| MCP moves to `@better-auth/mcp`                  | Update imports, endpoint paths, helper names, config shape, and schema.                                   |
| SAML defaults and config change                  | Confirm Node 20 or newer, update removed fields, update callback URLs, and review IdP-initiated flows.    |
| SCIM ownership becomes permanent                 | Reclaim old personal SCIM connections before regenerating their tokens.                                   |
| Proxy headers are not trusted by default         | Opt in with `advanced.trustedProxyHeaders: true` only when your proxy requires it.                        |
| Two-factor account lockout adds columns          | Run the schema migration if you use two-factor authentication.                                            |
| Custom adapters and storage need atomic methods  | Implement `incrementOne`, `increment`, or `consume`, depending on what you provide.                       |
| Captcha rules match full paths                   | Replace partial paths like `/sign-in` with `/sign-in/*` or `/sign-in/**`.                                 |
| Electron requires S256 PKCE                      | Upgrade the Electron client and server together.                                                          |

### Behavior and security changes

| Change                                                          | What changes                                                                                   |
| --------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `max_age` is enforced                                           | Clients that ask for fresh login now get it instead of being ignored.                          |
| Granted scopes are preserved                                    | Later logins no longer erase scopes granted earlier.                                           |
| Provider profile sync respects `input: false`                   | OAuth profile values no longer fill user fields that are marked server-only.                   |
| Google hosted-domain checks are applied to One Tap              | One Tap now follows the configured `hd` policy before creating a session.                      |
| SSO verifies every listed domain                                | Every comma-separated domain on a provider must prove ownership.                               |
| SAML validates audience, destination, and bearer recipient      | Responses that do not match the Service Provider are rejected before session creation.         |
| SAML Single Logout accepts only http(s) URLs                    | IdP logout form actions using schemes like `javascript:` or `data:` are rejected.              |
| SCIM write paths are stricter                                   | SCIM honors `active`, scopes deletes to the SCIM account, and rejects duplicate-email updates. |
| Server-side OAuth requests refuse redirects                     | Token, introspection, and JWKS requests no longer follow provider redirects.                   |
| Magic-link and email-OTP sign-in can clear unproven credentials | Proven mailbox control wins over an unconfirmed password on the same account.                  |
| Two-factor challenges cap wrong codes                           | Wrong TOTP and backup-code guesses are limited per sign-in challenge and per account.          |

### Deprecations and removals

| Change                                     | Replacement                          |
| ------------------------------------------ | ------------------------------------ |
| `oidcProvider` plugin removed              | Use `@better-auth/oauth-provider`.   |
| MCP plugin path moved out of `better-auth` | Use `@better-auth/mcp`.              |
| Generic OAuth client APIs changed          | Use the standard social client APIs. |

## Better Auth as an identity provider

This is the biggest area of the release. The OAuth provider grows from "it issues tokens" to "it behaves like a standards-based authorization server."

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Protected resources: one login server, many APIs

You can now describe each API behind your login server as its own resource. Each resource can have its own token lifetime, scopes, custom claims, and signing keys. Set them up with the `resources` option or the `oauthResource` admin API.

This replaces the old flat `validAudiences` list. The old shape had a real gap: a client could ask for a token aimed at any API on the list, even one the user never approved. Now the target API is captured at login time and locked to that grant. A later request can narrow the target API, but it cannot widen it.

This makes the common setup safer: one login server in front of several APIs, where a token minted for one API cannot be replayed against another.

Read the upgrade guide before deploying this if you use `@better-auth/oauth-provider`.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### DPoP: stolen tokens cannot be reused

Better Auth can now issue DPoP tokens. DPoP binds an access token to a key held by the client, so a copied token is not enough to call the API. This is the proof-of-possession model used by high-security OAuth deployments.

A client opts in at registration, on the login request, or by targeting an API that requires DPoP. The token stays bound to the client's key through refresh, introspection, and the userinfo endpoint. On the API side, one helper verifies the whole DPoP request, including replay protection across multiple server instances.

This needs a migration. The plain bearer-token helper is renamed, and the token tables get a token-binding column.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Single sign-out across apps

When a user signs out of your identity provider, every app they signed into through you can now be told to end its own session. This is OIDC Back-Channel Logout.

On sign-out, your server sends a signed logout message to each app that registered a logout URL. Tokens tied to that session stop working immediately. Before, they stayed valid until they expired.

Apps opt in by registering a `backchannel_logout_uri`. On serverless platforms, point `advanced.backgroundTasks.handler` at the runtime background-task API, such as Vercel `waitUntil` or Cloudflare `ctx.waitUntil`, so logout messages do not slow down sign-out.

<HeaderLabel variant="default">Behavior</HeaderLabel>

### Forced re-login actually works

OpenID has a `max_age` parameter that lets an app say "make the user log in again if their last login is older than this." Better Auth used to accept that request and ignore it.

Better Auth now honors `max_age`. If the user's login is too old, they are sent back to log in, and the new ID token shows the fresh login time. Apps that relied on the old ignored behavior may now see re-login prompts.

<HeaderLabel variant="default">Behavior</HeaderLabel>

### Token introspection is consistent

Introspection is how an API asks your login server whether a token is still active and what it can do. In 1.7 the answer is the same whether the token is an opaque string or a JWT. Opaque tokens used to return less information; now they return the same claims a JWT would for the same grant.

Your server also keeps control of standard claim names, so a custom-claims callback can no longer overwrite them by accident. A separate API service can now check a token issued to a different client, which is the normal setup where a front-end holds the token and a back-end API validates it.

<HeaderLabel variant="info">New</HeaderLabel>

### Refresh-token retries for native clients

The OAuth provider can now replay the same refresh-token response for duplicate refresh requests during `refreshTokenReuseInterval`. Strict replay handling remains the default. This option helps native and public clients recover when one local session rotates a refresh token and another local session retries with the old token.

MCP defaults this interval to 30 seconds for native and public clients. Set `refreshTokenReuseInterval: 0` to keep strict replay handling.

<HeaderLabel variant="info">New</HeaderLabel>

### An extension surface for new OAuth standards

OAuth keeps adding new standards, including token exchange, CIBA, and client-id metadata documents. Adding one used to mean changing the core OAuth provider. Now a companion plugin can register grant types, client-authentication methods, discovery metadata, and token claims through `extendOAuthProvider`.

The provider still owns security-sensitive decisions. A plugin can add to the discovery document and contribute claims, but it cannot override the server's own decision about a client.

<HeaderLabel variant="info">New</HeaderLabel>

### Client ID Metadata Documents

The new `@better-auth/cimd` package lets a client identify itself by hosting a small document at an HTTPS URL, where the URL itself is the client ID. This is how MCP clients can connect without being registered ahead of time.

The package includes safeguards for fetching remote documents: internal-network request protection, a fetch timeout, a response-size cap, and an `allowFetch` gate for your own allowlist.

<HeaderLabel variant="info">New</HeaderLabel>

### Self-service registration for machine clients

Dynamic Client Registration lets a client register itself instead of asking an admin to create it. In 1.7, a backend service can do this without a logged-in user by presenting a pre-shared registration token that you check with `validateInitialAccessToken`.

The registration endpoint also enforces the same permission checks as manual client-creation endpoints, and creating a client now returns the correct `201 Created` status.

<HeaderLabel variant="info">New</HeaderLabel>

### Working with real MCP clients

MCP clients like Claude, Codex, and Factory Droid register themselves in a way earlier versions rejected. Better Auth now accepts those registrations as public clients, which the standard allows, and tells the client which method it ended up with.

<HeaderLabel variant="info">New</HeaderLabel>

### Ask for specific user details

A client can now ask for individual user details by name through the standard `claims.userinfo` request parameter. The user-info endpoint returns the claims it can supply on top of the claims already covered by granted scopes. The request becomes part of the user's consent, and the provider advertises this support in discovery.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### ID-token claims stay protocol-safe

Custom ID-token claims can no longer overwrite protocol claims reserved for the server: issuer, subject, audience, expiry, nonce, session binding, `auth_time`, `acr`, `amr`, and `azp`. Your own namespaced claims still appear.

ID tokens now report `acr: "0"` instead of a vendor-specific value, and discovery no longer advertises authentication-strength classes the provider does not yet support. If a callback set one of the reserved claims, that value is now ignored.

<HeaderLabel variant="default">Security</HeaderLabel>

### Conformance polish

A broad set of fixes brings the OAuth and OpenID endpoints closer to the specs:

* **Standard error responses:** malformed requests return the `{ error, error_description }` shape the specs require.
* **No caching of credentials:** token, introspection, user-info, and registration responses tell proxies and browsers not to cache them.
* **Tamper-proof ID tokens:** ID tokens include an `at_hash` claim that binds them to their access token.
* **Stricter token handling:** refresh-token misuse, authorization-code replay, and `redirect_uri` mismatches are rejected with standard errors.
* **No server-side OAuth redirects:** token, introspection, and JWKS requests refuse provider redirects.
* **Flexible request handling:** authorization and user-info endpoints accept form-encoded requests, and userinfo accepts a bearer token in the request body.
* **PKCE flexibility:** confidential clients registered through Dynamic Client Registration can opt out of PKCE with `clientRegistrationRequirePKCE: false`; public clients and offline access still require it.

<HeaderLabel variant="warning">Deprecation</HeaderLabel>

### The old OIDC provider plugin is gone

The `oidcProvider` plugin, deprecated in 1.6, is removed. Move OpenID provider setups to `@better-auth/oauth-provider`.

## Connecting to more identity providers

The other half of the OAuth work is about Better Auth as the app receiving a login.

<HeaderLabel variant="default">Behavior</HeaderLabel>

### Scopes are kept across logins

When you ask a provider for permission to do something, that permission is a scope. Before 1.7, each new login overwrote the stored scopes with whatever that login asked for, so a permission granted earlier could disappear after a later login that asked for less access.

Better Auth now keeps the scopes already on the account across re-login and token refresh. Google also gets `includeGrantedScopes`, on by default, so each login carries forward the scopes already granted. Set it to `false` to request only the current login's scopes.

<HeaderLabel variant="info">New</HeaderLabel>

### Per-request login options

`signIn.social`, `linkSocial`, and `signIn.sso` now accept `additionalParams`, which adds extra options to one login URL. Before, these values could only be fixed server config, which made several integrations hard or impossible.

This unlocks:

* **Amazon Cognito:** pass Cognito's `identity_provider` value per login, so a user can go straight to the right upstream provider.
* **Microsoft Entra ID:** route users to the right tenant with `domain_hint`.
* **Google:** ask for offline access or ask for consent again on one login instead of every login.

Better Auth rejects parameters that callers must not control, such as `state` and `redirect_uri`.

<HeaderLabel variant="info">New</HeaderLabel>

### Certificate and signed-assertion login

Microsoft Entra ID can now authenticate with a signed certificate through `clientAssertion`. Generic OAuth providers can also authenticate to the token endpoint with a signed JWT through `tokenEndpointAuth`, or register as public clients with no secret.

Misconfigured signing setups now fail at startup instead of silently doing the wrong thing.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Generic OAuth is rebuilt

The generic OAuth plugin now uses the same path as built-in social providers. PKCE is on by default, issuer validation happens automatically through discovery, and the callback URL changes.

This changes the API you call and the callback URL you configure. The upgrade guide has the exact replacement steps.

<HeaderLabel variant="default">Security</HeaderLabel>

### Identity tokens are verified consistently

Point a generic OAuth provider at a discovery URL and Better Auth now verifies the provider identity token against that provider's published keys. Mobile and single-page apps can also use identity-token login through the same verifier.

PayPal now binds userinfo to the verified ID token subject during social sign-in. Custom providers should move from a `verifyIdToken` method to an `idToken` config.

<HeaderLabel variant="info">New</HeaderLabel>

### Providers that start the login themselves

Some providers, common in education, start login on their side and call your app back without the usual security token. Set `allowIdpInitiated: true` and Better Auth restarts the flow on the server with fresh security tokens, so protection is kept rather than turned off.

<HeaderLabel variant="info">New</HeaderLabel>

### Require a verified email per provider

You can now set `requireEmailVerification` on a single social or generic provider. When that provider reports an unverified email, the account is still created or linked, but no session is issued until the email is verified.

This is off by default and does not affect other providers. Turn it on only for providers whose email-verification signal you trust.

<HeaderLabel variant="default">Behavior</HeaderLabel>

### Provider profile fields respect server-only fields

OAuth sign-up and account-link profile sync now ignore provider profile values for user fields marked `input: false`. Input-allowed additional fields still persist from `mapProfileToUser`, and schema defaults still apply when OAuth creates a user.

If you used `mapProfileToUser` to fill `input: false` fields, move that assignment into server-side provisioning code.

<HeaderLabel variant="info">New</HeaderLabel>

### Account linking works in mobile browsers

Linking an anonymous account after a social login now works in Expo and other in-app browsers, where the login returns without the usual cookie. A new `addOAuthServerContext` API carries trusted data across the login that a client cannot forge.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Google One Tap signs in the right person

Google One Tap now verifies that the token is meant for your app and signs in the user who owns the Google account. Before, it matched on email, so a Google account could sign in a different user who happened to share that email.

A Google client ID is now required for One Tap. Hosted-domain restrictions also apply before One Tap creates a session, including the `hd: "*"` case for any Google Workspace hosted domain.

## Enterprise SSO, SAML, and SCIM

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Rotate SAML certificates without downtime

SAML signing certificates can now be a list, not just one value. An administrator can publish a new certificate next to the old one and finish the rotation without forcing everyone to sign in again.

The management endpoints now always return the certificate as a list, so code reading those endpoints must expect a list.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Stricter SAML validation

Several SAML checks that were not working before now work. Better Auth rejects responses whose audience, bearer recipient, or destination does not match the configured Service Provider. Unsolicited IdP-initiated logins are also off by default.

The underlying SAML library is upgraded to fix a high-severity vulnerability and now requires Node 20 or newer. Set `allowIdpInitiated: true` only if you depend on provider-started SAML login.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Simpler SAML configuration

SAML setup is simpler: the callback URL is derived automatically, service-provider metadata is generated for you, and several unused fields are removed. Single Logout also correctly ends the user's session now.

A few fields and one endpoint path change. See the upgrade guide before deploying SAML changes.

<HeaderLabel variant="default">Security</HeaderLabel>

### SSO domain verification checks every domain

When an SSO provider lists multiple comma-separated domains, every listed domain must now publish the verification TXT record before the provider is marked verified. This prevents one verified domain from covering another unverified domain in the same provider config.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### SCIM groups and ownership

SCIM gains durable group resources with membership, roles, and lifecycle endpoints.

A security fix ships with it: a personal SCIM connection is now always owned by the user who created it. Before, ownership was optional, and an unowned connection could be read or deleted by any signed-in user.

This needs a migration and one manual cleanup. Connections created before the upgrade need to be deleted or re-owned before you regenerate their tokens.

<HeaderLabel variant="default">Behavior</HeaderLabel>

### SCIM write paths are stricter

SCIM now honors the `active` attribute on create, update, and patch. Setting `active: false` deactivates the user through the admin plugin and revokes their sessions. SCIM email changes now reject duplicate emails with `409`.

Non-organization SCIM deletes also unlink the SCIM provider account when the user has other linked identities, and delete the global user only when the SCIM account is their sole identity.

<HeaderLabel variant="info">New</HeaderLabel>

### OpenID SSO on Cloudflare Workers

OpenID SSO with discovery now works on Cloudflare Workers. Endpoints that redirect are rejected with a clear configuration error instead of failing in a runtime-specific way.

## MCP

<HeaderLabel variant="warning">Breaking</HeaderLabel>

The MCP plugin moves into its own package, `@better-auth/mcp`, built on the OAuth provider. MCP now inherits the same authorization-server behavior as `@better-auth/oauth-provider`.

The move changes imports, endpoint paths, helper names, database tables, and config shape. It requires the `jwt()` plugin and an explicit resource identifier. Client adapters ship for Hono, the official MCP SDK, and `mcp-use`.

This is the largest single change for existing MCP users. The upgrade guide has a step-by-step migration.

## Sessions

<HeaderLabel variant="info">New</HeaderLabel>

* **Verify sessions with a public key:** the session cookie cache can use asymmetric JWTs, so services can read sessions with a public key instead of sharing the signing secret.
* **No more first-render flash:** `hydrateSession` seeds the client with a server-fetched session, so `useSession` has data on the first render.
* **Larger caches survive:** session cookies near the browser size limit are split into chunks instead of being silently dropped.

<HeaderLabel variant="default">Behavior</HeaderLabel>

* **Tokens revoked on sign-out:** signing out now revokes OAuth tokens tied to that session, even when session data lives in a separate store.

## Secure by default

<HeaderLabel variant="default">Security</HeaderLabel>

### Atomic state, no more races

The largest security change replaces "check, then write" database logic with single atomic operations. These bugs only show up under load: two requests using the same one-time token both succeed, a team admits more members than its limit, or a rate limit slips when many requests arrive at once.

Normal Better Auth users get the fix automatically:

* **One-time tokens** for password reset, email and phone codes, device codes, magic links, and more can no longer be used twice, even by simultaneous requests.
* **Team limits** are enforced correctly under load.
* **Rate limits** are checked and counted in a single step.
* **Breached-password checks** cover more password-setting routes by default.

If you wrote a custom database adapter or storage backend, this introduces required methods. See the upgrade guide.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Forwarded proxy headers are not trusted by default

When your app reads its own address from the request, it now uses the `Host` header and ignores forwarded headers (`x-forwarded-host`, `x-forwarded-proto`) unless you opt in with `advanced.trustedProxyHeaders: true`.

Common platforms such as nginx, Vercel, Cloudflare, and Netlify rewrite the host for you and usually need no change. Custom proxy setups that only forward the public host through forwarded headers should opt in explicitly.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Electron requires modern PKCE

The Electron login flow now requires S256 PKCE and no longer trusts a custom header to set the request origin. The client and server must be upgraded together. Custom URL-scheme matching is also stricter, so a trusted entry cannot be copied by a lookalike host.

<HeaderLabel variant="warning">Breaking</HeaderLabel>

### Captcha matches full paths

Captcha rules now match full request paths or explicit wildcards. Replace a partial path like `/sign-in` with `/sign-in/*` or `/sign-in/**`.

<HeaderLabel variant="info">New</HeaderLabel>

### A gate for new identities

The new `user.validateUserInfo` hook lets your app reject an identity before a user is created or an account is linked, across every sign-up method. It also runs when an existing user signs in again through a provider, so a policy can catch an identity that has moved out of bounds.

<HeaderLabel variant="default">Security</HeaderLabel>

### Two-factor and passwordless hardening

Two-factor verification now has both challenge-level and account-level limits. TOTP and backup-code challenges stop after five wrong codes, and the account-level lockout applies across TOTP, email OTP, and backup codes. By default, an account locks for 15 minutes after 10 consecutive failed verifications.

Magic-link and email-OTP sign-in now treat proven mailbox control as the source of truth when an account email had never been confirmed. If that account had an unproven password, Better Auth removes the password and revokes existing sessions before signing the user in.

## Developer experience

<HeaderLabel variant="info">New</HeaderLabel>

* **i18n in 22 languages:** `@better-auth/i18n` ships built-in translations for 22 languages, and the English fallback is fixed.
* **A `create-admin` CLI command:** create your first admin user from the CLI.
* **Cloudflare and Expo fixes:** Cloudflare Workers apps no longer crash on startup when importing certain subpaths, and Expo no longer loses logins from providers that issue large tokens.
* **A server-only phone OTP API:** verify and consume a code for custom flows, without touching users or sessions.
* **An immutable username option:** let a user set a username once and not change it.
* **Smaller quality-of-life fixes:** clear an organization's logo with `logo: null`, list another member's teams, support Stripe SDK v21 and v22, and use a real-HTTP test helper.

## New integrations unlocked

These setups were impossible, broken, or unsafe before, and work now.

### Logging your users in with an external provider

| Provider or setup                                         | What was blocked                                                      | How it works now                                                         |
| --------------------------------------------------------- | --------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Amazon Cognito** federated login                        | You could not route a user to a specific upstream provider per login. | Use typed `identityProvider` and per-request `additionalParams`.         |
| **Microsoft Entra ID** certificate login                  | Only shared-secret login was supported.                               | Use `clientAssertion` for certificate login and `domain_hint` per login. |
| **Google** offline and incremental access                 | Options were global, and granted scopes were overwritten each login.  | Use per-request options, preserved scopes, and `includeGrantedScopes`.   |
| **Any OpenID provider** via discovery                     | The provider identity token was not verified.                         | Point at a discovery URL; tokens are verified automatically.             |
| **Zitadel, Auth0, and other multi-tenant OIDC providers** | No way to send extra parameters when refreshing a token.              | Use `refreshTokenParams` on refresh, without a full redirect.            |
| **Providers needing signed-assertion login**              | Only secret-based login was supported.                                | Use `tokenEndpointAuth` with a signed JWT.                               |
| **Clever and similar education providers**                | A login started by the provider failed.                               | Use `allowIdpInitiated: true` to restart the flow safely.                |
| **Keycloak and other large-token providers on Expo**      | The login cookie was silently dropped on the device.                  | The Expo store splits and reassembles it.                                |

### Other apps logging in through you

| Setup                                          | What was blocked                        | How it works now                                                           |
| ---------------------------------------------- | --------------------------------------- | -------------------------------------------------------------------------- |
| **APIs needing theft-resistant tokens**        | Only plain bearer tokens.               | Use DPoP tokens bound to the client's key.                                 |
| **Several APIs behind one login server**       | One token could be used on any API.     | Use per-API resources; tokens are locked to the API they were issued for.  |
| **Machine clients registering themselves**     | Registration required a logged-in user. | Use pre-shared registration tokens.                                        |
| **MCP clients (Claude, Codex, Factory Droid)** | Their registration was rejected.        | They are accepted automatically, with Client ID Metadata Document support. |
| **Single sign-out across apps**                | Sign-out did not notify other apps.     | Use OIDC Back-Channel Logout.                                              |
| **Forced re-login**                            | The request was ignored.                | `max_age` is enforced.                                                     |

### Enterprise SSO

| Setup                                | What was blocked                      | How it works now                                    |
| ------------------------------------ | ------------------------------------- | --------------------------------------------------- |
| **SAML certificate rotation**        | Only one certificate was accepted.    | A list of certificates is accepted during rotation. |
| **SCIM group provisioning**          | No durable group resources.           | Groups have first-class lifecycle endpoints.        |
| **OpenID SSO on Cloudflare Workers** | Redirecting endpoints broke the flow. | They fail with a clear configuration error.         |

## Upgrading

Most projects upgrade with one command:

```package-install
npx auth upgrade
```

Several features add or change database tables, and a few need a manual data step. Read the [1.7 upgrade guide](/docs/guides/1-7-upgrade-guide) for the changes that apply to your setup and the exact steps to take.
